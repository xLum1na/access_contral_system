# 系统架构设计文档

## 1.整体架构

设备层 → 网络接入层 → 云端核心服务层 → 平台集成层 → 用户层。

* **设备层**：物理硬件终端，包括门禁主机（人脸识别/显示）、继电器控制、报警模块及边缘网关等。负责数据采集与物理执行。
* **网络接入层**：负责设备与云端的安全连接。支持 Wi-Fi、以太网等局域通信，通过边缘网关或直连方式接入互联网。
* **云端核心服务层**：系统的“大脑”，负责设备管理、权限下发、业务逻辑处理、数据存储与分析。采用微服务架构，保障高并发与高可用。
* **平台集成层**：提供标准化的 OpenAPI 和 Webhook，用于与企业现有系统（如 HR 考勤系统、OA 办公系统、智慧园区平台、公安安防平台）进行数据互通与集成。
* **用户层**：面向不同角色的终端交互界面，包括物业管理 Web 后台、员工/住户移动端 APP/小程序、以及访客预约 H5 页面。

## 2.固件架构设计图

![](/home/xiamu/git/project/access_contral_system_project_v1.0/documents/软件架构.png)

门禁设备固件采用分层模块化设计，以支持跨硬件平台的移植和功能的快速迭代：

* **硬件抽象层 (PAL)**：屏蔽不同厂商库、底层芯片差异，提供统一的 GPIO、I2C、SPI、UART、Flash 等硬件接口调用，接口格式统一为pal_xxx_xxx()，例如：pal_gpio_set_level()。

- **OS抽象层 (OSAL)**：适配不同的操作系统（如 FreeRTOS、RT-Thread 或嵌入式 Linux），提供统一的任务调度、内存管理、信号量和消息队列接口，接口统一为osal_xxx_xxx()，例如：osal_vtaskdelay()。
- **设备驱动层(DAL)**：包含各类外设驱动，如网络芯片驱动、摄像头/ISP驱动、音频Codec驱动、继电器/门锁控制驱动等，统一接口为dal_xxx_xxx()，例如：dal_dispaly_draw_bitmap()。
- **服务层**：提供系统级中间件服务，包括网络协议栈（TCP/IP, TLS）、MQTT客户端、本地数据库（SQLite）、OTA升级服务、音视频流媒体处理及国密/AES加密服务。
- **应用层**：实现具体业务逻辑，包括身份鉴权（人脸比对）、门禁控制状态机、离线记录缓存、防拆报警逻辑、设备UI交互（针对带屏设备）等。



## 3.网络接入与通信架构

本部分定义了设备与云端、设备与设备之间的通信机制，确保数据在弱网或断网环境下的可靠性。

### 3.1 通信协议选型

- **MQTT 协议**：作为设备与云端的核心长连接协议。用于设备状态心跳上报、远程开门指令下发、权限变更实时推送。具有低带宽、低功耗的特点。
- **HTTPS/RESTful API**：用于设备首次激活、配置拉取、固件 OTA 升级包下载、以及大文件（如人脸底库图片、批量日志）的上传与下载。
- **WebRTC / RTSP**：针对可视对讲门禁，用于音视频流的实时传输，支持 P2P 穿透或 TURN 服务器中继。
- **局域网协议 (RS485 / Wiegand / OSDP)**：用于门禁主机与子设备（如独立读卡器、出门按钮、电锁）之间的底层硬件通信。

### 3.2 弱网与离线机制（边缘计算）

- **离线鉴权**：云端定期将权限名单（人脸特征值、卡号）下发至设备本地存储。当网络中断时，设备自动切换至本地鉴权模式，确保门禁正常开关。
- **断网续传**：设备在离线状态下产生的开门记录、报警日志会暂存于本地 Flash/SD 卡中，网络恢复后自动按时间戳顺序补传至云端。

### 3.3 安全与加密

- **传输加密**：所有公网通信强制采用 TLS/SSL 加密（MQTTS / HTTPS）。
- **设备认证**：采用“一机一密”或 X.509 证书机制，设备接入云端 IoT 网关时需进行双向身份认证，防止伪造设备接入。

## 4.云端核心服务架构

云端采用**微服务架构（Microservices）**，基于 Spring Cloud / Kubernetes 构建，各服务独立部署、独立扩展。

### 4.1 核心微服务划分

- **IoT 设备接入服务**：基于 EMQX / Mosquitto 等高性能 MQTT Broker，负责维持海量设备的长连接，处理消息路由与协议解析。
- **设备管理服务**：负责设备注册、生命周期管理、固件版本控制、OTA 升级任务调度及远程配置下发。
- **权限与人员管理服务**：管理组织架构、人员信息、角色权限。负责将人员权限转化为设备可识别的“权限组”，并下发至对应设备。
- **门禁与考勤业务服务**：处理开门请求、生成通行记录、计算考勤报表、处理访客预约审批流程。
- **告警与消息服务**：处理设备防拆、门长时间未关、非法闯入等告警事件，并通过短信、APP 推送、微信模板消息触达相关人员。

### 4.2 数据存储架构

- **关系型数据库 (MySQL / PostgreSQL)**：存储核心业务数据，如人员信息、设备台账、权限配置、系统字典等。
- **时序数据库 (TDengine / InfluxDB)**：存储设备上报的海量时序数据，如心跳日志、传感器状态、温湿度数据。
- **缓存与消息队列 (Redis / RabbitMQ / Kafka)**：Redis 用于高频数据缓存（如 Token、设备在线状态）；Kafka/RabbitMQ 用于服务间的异步解耦和削峰填谷（如海量通行记录的异步落库）。
- **对象存储 (OSS / MinIO)**：存储人脸抓拍图片、访客证件照、固件安装包及系统日志文件。

## 5.APP端架构

移动端（APP/小程序）采用前后端分离设计，支持 iOS、Android 及微信小程序多端运行（可基于 Flutter、React Native 或 Uni-app 开发）。

### 5.1 客户端分层架构

- **UI 表现层**：负责界面渲染与用户交互，采用组件化设计，提供统一的 UI 规范（如扫码开门、蓝牙开门、访客邀请、消息中心）。

- **业务逻辑层**：处理核心业务流程，如身份认证、权限校验、本地蓝牙/Wi-Fi 通信逻辑、消息推送处理。

- 网络与数据层

  ：

  - **网络请求**：基于 Retrofit/Axios 封装统一的 HTTP 请求拦截器，处理 Token 刷新、签名校验、异常重试。
  - **本地存储**：使用 SQLite 或 MMKV 缓存用户配置、未读消息及离线必要的门禁密钥。

### 5.2 核心功能模块

- 多模式开门：
  - **远程开门**：通过云端下发指令（需设备在线）。
  - **蓝牙开门**：APP 通过 BLE 协议直连门禁设备，进行近距离加密鉴权开门（无需互联网）。
  - **二维码开门**：APP 生成动态加密二维码，设备扫码解析鉴权。
- **可视对讲**：集成音视频 SDK，当访客呼叫时，APP 弹出接听界面，支持双向语音与视频预览。
- **访客管理**：住户可通过 APP 生成带时效的“访客通行码”或下发临时人脸权限，分享给微信好友。
- **消息与告警**：接收云端推送的开门通知、低电量告警、门未关严提醒。

### 5.3 安全机制

- **数据脱敏**：本地不存储明文密码，敏感信息（如人脸数据）不在本地留存。
- **动态令牌**：蓝牙开门和二维码开门采用动态时间戳 + 非对称加密签名，防止抓包重放攻击。
- **环境检测**：APP 启动时检测设备运行环境，防范 Root/越狱设备或模拟器带来的安全风险。

## 6.安全架构

本系统的安全架构贯穿“端-管-云-用”全链路，确保物理设备防破坏、网络传输防篡改、云端数据防泄露，并严格满足个人信息保护与隐私合规要求。

### 6.1 设备与物理安全

- **硬件防拆与报警**：门禁主机内置防拆微动开关与陀螺仪，设备被非法拆卸、撬动或暴力破坏时，立即触发本地声光报警，并向云端实时上报防拆告警事件。
- **安全芯片 (SE/TEE)**：核心设备集成硬件安全芯片或采用可信执行环境（TEE），用于安全存储设备证书、私钥及核心加密算法，防止密钥被物理提取或内存dump。
- **固件与启动安全**：支持安全启动（Secure Boot），在设备上电时校验 Bootloader 和固件的数字签名，防止设备被刷入恶意或篡改的固件。

### 6.2 网络与通信安全

- **全链路加密**：所有端到云通信（MQTT/HTTP）强制采用 TLS 1.2/1.3 加密，防止数据在公共网络传输过程中被窃听或中间人篡改。
- **防重放与防伪造**：控制指令下发与设备数据上报均携带时间戳与随机数（Nonce），并结合 HMAC 签名校验，有效防止网络抓包重放攻击。
- **网络隔离与防护**：云端部署于 VPC（虚拟私有云），设备接入网关、核心业务微服务与数据库处于不同的安全子网。通过严格的安全组策略、WAF（Web应用防火墙）限制跨网段访问与恶意流量。

### 6.3 云端与应用安全

- **身份与访问管理 (IAM)**：采用 OAuth 2.0 / JWT 标准进行用户身份认证，结合 RBAC（基于角色的访问控制）模型，实现细粒度的菜单、按钮与数据级权限管控。
- **接口安全**：对外开放的 OpenAPI 采用 AppKey/AppSecret 签名机制，并配置 IP 白名单与请求频率限制（Rate Limiting），防范恶意调用、越权访问与 CC 攻击。
- **安全审计**：记录所有后台管理员与系统的关键操作日志（包括登录、配置修改、权限下发、远程开门等），确保所有高危操作可追溯、可审计。

### 6.4 数据隐私与合规

- **生物特征保护**：云端与设备端**不存储原始人脸图片**用于日常比对，仅存储不可逆的人脸特征值（Feature Vector）。原始图片仅在用户首次注册或授权时加密存储于独立的 OSS 隔离桶中。
- **数据脱敏与加密**：数据库中的敏感信息（如手机号、身份证号、密码）采用 AES-256 加密存储，并在前端展示时进行掩码脱敏处理（如 `138****1234`）。
- **合规性设计**：系统架构设计遵循《个人信息保护法》(PIPL) 及 GDPR 等相关法规，提供用户数据导出与“被遗忘权”（数据彻底销毁）的功能接口。

## 7.数据架构

数据架构关注数据从采集、传输、存储到处理、应用的全生命周期管理，旨在挖掘通行数据的业务价值，并保障数据的高效流转与高可用性。

### 7.1 核心数据模型

系统核心数据实体主要包括：

- **主数据 (Master Data)**：组织架构、人员信息、设备台账、区域与门禁点配置、角色与权限组。
- **交易数据 (Transaction Data)**：通行记录（刷卡/人脸/远程开门）、访客预约记录、设备告警事件、系统操作日志。
- **时序数据 (Time-Series Data)**：设备心跳状态、门锁状态、传感器数据（如门磁、温湿度）、网络信号强度。

### 7.2 数据采集与流转架构

- **端侧采集**：设备通过传感器和摄像头采集物理状态与生物特征，经过边缘计算（如本地人脸比对）后，将结构化结果打包上报。

- **消息路由与削峰**：设备上报的海量通行记录和心跳数据，首先接入 MQTT Broker，随后通过规则引擎路由至 Kafka 消息队列进行削峰填谷，避免突发流量冲垮后端数据库。

- 流批处理：

  - **实时流处理 (Flink/Spark Streaming)**：实时消费 Kafka 数据，用于触发即时告警（如黑名单人员闯入）、更新设备在线状态、推送 APP 消息。
- **离线批处理**：定时将数据从消息队列或操作数据存储（ODS）抽取至数据仓库，进行数据清洗、去重和聚合计算。

### 7.3 数据存储与治理

- 冷热数据分离：

  - **热数据**（近3个月的通行记录、实时状态）存储于 MySQL/PostgreSQL 及 Redis 中，保障高频查询的低延迟。
  - **冷数据**（历史通行记录、历史日志）定期归档至低成本的对象存储或 HBase/ClickHouse 中，满足长期合规留存要求。
  
- **数据质量与治理**：建立数据字典与元数据管理，对异常数据（如时间戳错乱、重复上传的记录）进行自动过滤与修正，确保报表统计的准确性。

### 7.4 数据分析与应用

- **BI 报表与大屏**：基于清洗后的数据，提供多维度的可视化报表，包括人流量趋势、各部门考勤统计、设备在线率、告警类型分布等，支持物业管理大屏展示。

- AI 智能分析：

  - **通行轨迹追踪**：结合多个门禁点的通行记录，还原特定人员在园区/楼宇内的活动轨迹。
  - **异常行为预警**：通过算法识别异常模式，如“尾随通行”（一次刷卡多人通过）、“非工作时间频繁出入”、“长期未归”等，并自动生成安防预警。
  
- **数据开放**：通过标准化的数据订阅服务（Data Subscription），将清洗后的通行与考勤数据实时推送给第三方 HR 或 OA 系统，打破数据孤岛。

## 8.部署与运维架构

本系统采用云原生（Cloud Native）理念进行部署与运维设计，结合自动化流水线与全链路可观测性体系，保障系统的高可用性、快速迭代能力以及故障的快速定位与恢复。

### 8.1 云原生部署架构

- **容器化与编排**：所有云端微服务均打包为 Docker 镜像，基于 Kubernetes (K8s) 进行容器编排与调度，实现服务的弹性伸缩（HPA）与故障自愈。：
  - **多可用区部署**：核心服务与数据库跨可用区（AZ）部署，避免单机房故障导致系统瘫痪。
  - **负载均衡**：通过云厂商 LB 或 Ingress Nginx 进行流量分发，消除单点故障。
  - **数据库高可用**：关系型数据库采用主从架构（如 MySQL MHA/Orchestrator），Redis 采用 Cluster 集群模式，MQTT Broker 采用多节点集群部署。

### 8.2 CI/CD 持续集成与交付

- **自动化流水线**：基于 GitLab CI / Jenkins 构建标准化的 DevOps 流水线。代码提交后自动触发代码规范检查（SonarQube）、单元测试、镜像构建与漏洞扫描。
- **灰度发布与回滚**：支持蓝绿部署与金丝雀发布（Canary Release）。新版本上线时先导入少量流量或特定租户，观察无异常后再全量放开；若出现严重问题，支持一键回滚至上一稳定版本。

### 8.3 全链路可观测性体系

- **指标监控 (Metrics)**：使用 Prometheus 采集 K8s 节点、微服务 JVM/Go 运行时、MQTT Broker 连接数及业务指标，通过 Grafana 构建可视化监控大屏。
- **日志中心 (Logging)**：采用 EFK/ELK (Elasticsearch, Fluentd/Filebeat, Kibana) 架构，统一收集云端微服务日志、Nginx 访问日志以及设备端上报的错误日志，支持海量日志的秒级检索。
- **链路追踪 (Tracing)**：引入 SkyWalking 或 Jaeger，为每个请求生成全局 TraceID，贯穿 APP、网关、微服务与数据库，快速定位慢查询与跨服务调用瓶颈。
- **智能告警**：通过 Alertmanager 配置告警规则（如 CPU 飙升、MQTT 掉线率异常、错误日志激增），联动钉钉、企业微信或短信/电话自动通知值班运维人员。

### 8.4 设备端远程运维

- **OTA 灰度升级**：支持按设备型号、固件版本、特定区域或百分比进行 OTA 固件灰度下发，升级失败自动回滚至旧版本。
- **远程诊断与调试**：在设备授权与安全认证的前提下，支持建立加密的远程 SSH/Tunnel 通道，或远程下发指令拉取设备本地运行日志，减少现场实施与运维成本。

## 9.测试架构

针对门禁物联网系统“软硬结合、高并发、重安全”的特点，测试架构不仅涵盖传统的软件分层测试，还引入了硬件在环（HIL）与 IoT 专项测试，确保系统在复杂物理环境与极端网络条件下的可靠性。

### 9.1 自动化测试分层策略

- **单元测试**：开发人员编写，覆盖核心业务逻辑、加密算法、权限校验等底层代码，要求核心模块代码覆盖率达到 80% 以上。
- **接口与集成测试**：基于 Postman/JMeter 或自动化测试框架（如 PyTest/REST Assured），对云端 OpenAPI、微服务间 RPC 调用进行自动化契约测试与业务流测试。
- **UI 自动化测试**：针对 Web 管理后台和移动端 APP，使用 Selenium/Appium 进行核心主流程（如人员下发、远程开门）的自动化回归测试。

### 9.2 IoT 与硬件专项测试

- **设备模拟与仿真测试**：编写 MQTT 模拟器脚本，批量虚拟数万台门禁设备接入云端，验证 IoT 接入网关的连接承载力、消息路由准确性及权限下发的吞吐量。
- **硬件在环测试 (HIL)**：搭建自动化测试工装，将真实门禁主板与测试仪器连接。通过脚本自动控制继电器输入/输出、模拟刷卡/人脸信号、触发防拆开关，验证固件底层逻辑与硬件交互的正确性。
- **弱网与异常环境测试**：使用网络损伤仪或弱网模拟工具（如 Clumsy/ATC），在丢包、高延迟、网络频繁闪断等恶劣网络环境下，测试设备的断网续传、离线鉴权及重连机制。

### 9.3 性能与压力测试

- **高并发压测**：针对早晚高峰期“集中打卡/通行”场景，使用 JMeter/LoadRunner 对云端核心接口（如鉴权、记录上报）进行压测，评估系统的 TPS、响应时间（RT）及资源消耗，找出性能瓶颈并进行调优。
- **长连接稳定性测试**：验证 MQTT Broker 在维持海量设备长连接（如 10万+ 并发连接）情况下的内存占用、心跳处理能力及消息不丢失的可靠性。

### 9.4 安全与合规测试

- **渗透测试**：定期邀请内部安全团队或第三方安全机构，对 Web 端、APP 端及 OpenAPI 进行黑盒/白盒渗透测试（如 SQL 注入、XSS、越权访问）。
- **固件与接口抓包分析**：对设备与云端的通信进行抓包，验证 TLS 加密的有效性、防重放机制的严密性，以及是否存在敏感数据明文传输。
- **合规性审查**：针对 APP 隐私政策、权限索取、人脸数据采集与存储流程进行合规性走查，确保符合应用市场上架要求及国家个人信息保护法规。
