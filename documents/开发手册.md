## 1 固件开发

### 1.1 架构总览

固件采用严格的分层架构，自底向上分为五层：

```
┌──────────────────────────────────────────────┐
│  APP 层 (main.c)                             │
│  应用初始化、业务流程编排                      │
├──────────────────────────────────────────────┤
│  UI 层 (components/ui)                       │
│  图形界面框架封装（预留）                      │
├──────────────────────────────────────────────┤
│  SERVICE 层 (components/service)             │
│  数据库管理 (用户/人脸特征/日志)               │
├──────────────────────────────────────────────┤
│  DAL 层 (components/dal)                     │
│  显示/触摸/音频/网络/存储/PIR/继电器           │
├──────────────┬───────────────────────────────┤
│  PAL 层      │  OSAL 层                      │
│  GPIO/I2C/   │  Task/Mutex/Semaphore/        │
│  SPI/UART/   │  Queue/Timer/Memory           │
│  MIPI_DSI/   │                               │
│  LEDC/LOG    │                               │
├──────────────┴───────────────────────────────┤
│  ESP-IDF + FreeRTOS                          │
│  (driver, esp_lcd, esp_video, esp_eth, …)    │
└──────────────────────────────────────────────┘
```

**分层原则：**

| 层次 | 职责 | 依赖 |
|------|------|------|
| PAL | 封装 ESP-IDF 硬件驱动，提供统一接口 | ESP-IDF 驱动 |
| OSAL | 封装 FreeRTOS 内核对象，便于跨平台移植 | FreeRTOS |
| DAL | 完整的设备驱动（含外部芯片初始化逻辑） | PAL / OSAL |
| SERVICE | 业务数据管理，不直接操作硬件 | DAL (存储) |
| UI | 图形界面框架封装 | DAL (显示/触摸) |
| APP | 系统初始化、业务编排 | 全部下层 |

**关键约束：**
- 上层可调用下层，下层不可反向依赖上层
- 同层模块间尽量避免直接依赖（PAL 中仅 `pal_mipi_dsi` → `pal_ledc` 为例外）
- 应用代码不可直接调用 ESP-IDF API，必须通过 PAL/OSAL 间接访问

### 1.2 平台抽象层 (PAL)

PAL 将 ESP-IDF 硬件驱动封装为统一接口，所有函数返回 `0` 成功，负数错误码。

#### 1.2.1 GPIO — `pal_gpio`

**文件：** `components/pal/pal_gpio/`

**功能：** 引脚输入/输出、内部上下拉、外部中断。

**主要 API：**

```c
int pal_gpio_set_direction(int pin, pal_gpio_dir_t dir);
int pal_gpio_set_pull_mode(int pin, pal_gpio_pull_t pull);
int pal_gpio_write(int pin, int level);
int pal_gpio_read(int pin);
int pal_gpio_toggle(int pin);
int pal_gpio_install_isr_service(int flags);
int pal_gpio_set_intr(int pin, pal_gpio_intr_edge_t edge, pal_gpio_isr_cb_t cb, void *arg);
int pal_gpio_remove_intr(int pin);
```

**中断触发边沿：** `PAL_GPIO_INTR_DISABLE` / `POSEDGE` / `NEGEDGE` / `ANYEDGE` / `LOW_LEVEL` / `HIGH_LEVEL`

> **注意：** ISR 回调应保持极短（仅设标志位或发队列），复杂处理放在任务上下文。

#### 1.2.2 I2C — `pal_i2c`

**文件：** `components/pal/pal_i2c/`

**功能：** I2C 主机总线及从设备管理。

**主要 API：**

```c
int pal_i2c_bus_init(pal_i2c_bus_handle_t *handle, const pal_i2c_bus_config_t *cfg);
int pal_i2c_bus_deinit(pal_i2c_bus_handle_t *handle);
int pal_i2c_dev_probe(pal_i2c_bus_handle_t bus, uint16_t addr);
int pal_i2c_dev_attach(pal_i2c_dev_handle_t *dev, pal_i2c_bus_handle_t bus, const pal_i2c_dev_config_t *cfg);
int pal_i2c_dev_detach(pal_i2c_dev_handle_t *dev);

/* 数据传输 */
int pal_i2c_write(pal_i2c_dev_handle_t dev, const uint8_t *data, size_t len);
int pal_i2c_read(pal_i2c_dev_handle_t dev, uint8_t *data, size_t len);
int pal_i2c_write_reg(pal_i2c_dev_handle_t dev, uint8_t reg_addr, const uint8_t *data, size_t len);
int pal_i2c_read_reg(pal_i2c_dev_handle_t dev, uint8_t reg_addr, uint8_t *data, size_t len);

/* 便捷单字节操作 */
int pal_i2c_write_reg_byte(pal_i2c_dev_handle_t dev, uint8_t reg, uint8_t val);
int pal_i2c_read_reg_byte(pal_i2c_dev_handle_t dev, uint8_t reg, uint8_t *val);
```

**当前硬件连接（ESP32-P4）：**

| 信号 | GPIO | 备注 |
|------|------|------|
| SDA | GPIO7 | OV5647 + ATtiny88 共用 |
| SCL | GPIO8 | OV5647 + ATtiny88 共用 |
| 频率 | 100 kHz | SCCB 标准速率 |

> **共用总线设计：** 摄像头 (OV5647 @ 0x36) 和显示背光 (ATtiny88 @ 0x45) 复用同一 I2C 总线。`pal_i2c_get_bus_handle()` 可暴露底层句柄供 `esp_video_init` 等外部框架复用，避免重复创建。

#### 1.2.3 MIPI DSI — `pal_mipi_dsi`

**文件：** `components/pal/pal_mipi_dsi/`

**功能：** ESP32-P4 内置 MIPI DSI 主机 + DPI 面板控制器驱动。管理 DSI 总线、DPI 时序、帧缓冲、背光 PWM。

**主要 API：**

```c
int pal_mipi_dsi_init(pal_mipi_dsi_handle_t *handle, const pal_mipi_dsi_config_t *cfg);
int pal_mipi_dsi_deinit(pal_mipi_dsi_handle_t *handle);
int pal_mipi_dsi_display_on(pal_mipi_dsi_handle_t handle);
int pal_mipi_dsi_display_off(pal_mipi_dsi_handle_t handle);
int pal_mipi_dsi_draw_bitmap(pal_mipi_dsi_handle_t handle, uint16_t x, uint16_t y,
                             uint16_t w, uint16_t h, const void *data);
int pal_mipi_dsi_fill(pal_mipi_dsi_handle_t handle, uint16_t x, uint16_t y,
                      uint16_t w, uint16_t h, uint32_t color);
int pal_mipi_dsi_get_fb(pal_mipi_dsi_handle_t handle, void **fb0, void **fb1);
int pal_mipi_dsi_set_backlight(pal_mipi_dsi_handle_t handle, uint8_t percent);
int pal_mipi_dsi_send_generic_write(pal_mipi_dsi_handle_t handle, uint8_t vc,
                                    const uint8_t *payload, size_t len);
```

**支持像素格式：**
- DSI 输出：`PAL_DSI_COLOR_RGB565` / `RGB666` / `RGB888`
- 帧缓冲输入：`PAL_DSI_IN_COLOR_RGB565` / `RGB888`

**当前显示配置：**

| 参数 | 值 | 说明 |
|------|-----|------|
| 分辨率 | 800×480 | 树莓派 7" DSI |
| DSI 通道数 | 1-lane | 600 Mbps |
| DPI 时钟 | 25.98 MHz | 面板规格 |
| 像素格式 | RGB888 | 与 ISP 输出零拷贝匹配 |
| 帧缓冲数 | 1 | 双缓冲在 TC358762 上有切换干扰 |

### 1.3 操作系统抽象层 (OSAL)

OSAL 封装 FreeRTOS 内核对象，将 ESP-IDF 特定 API 转换为通用接口。所有句柄类型均为 `void *`。

**统一头文件：** `components/osal/include/osal.h`（包含全部子模块）

#### 1.3.1 任务管理 — `osal_task`

```c
int osal_task_create(const char *name, osal_task_func_t func, void *arg,
                     uint32_t stack_size, uint32_t priority, int core_id, osal_task_handle_t *task);
int osal_task_delete(osal_task_handle_t task);
int osal_task_delay_ms(uint32_t ms);
int osal_task_get_stack_high_water_mark(osal_task_handle_t task);
```

> **栈水位监控：** 使用 `osal_task_get_stack_high_water_mark()` 定期检查各任务最小剩余栈空间，及时发现栈溢出风险。

#### 1.3.2 互斥锁 — `osal_mutex`

```c
int osal_mutex_create(osal_mutex_handle_t *mutex);
int osal_mutex_lock(osal_mutex_handle_t mutex, uint32_t timeout_ms);
int osal_mutex_unlock(osal_mutex_handle_t mutex);
int osal_mutex_delete(osal_mutex_handle_t mutex);
```

> 实现为 FreeRTOS 递归互斥锁（优先级继承），适合短临界区保护。

#### 1.3.3 信号量 — `osal_semaphore`

```c
int osal_sem_create_binary(osal_sem_handle_t *sem);
int osal_sem_create_counting(osal_sem_handle_t *sem, uint32_t max, uint32_t init);
int osal_sem_take(osal_sem_handle_t sem, uint32_t timeout_ms);
int osal_sem_give(osal_sem_handle_t sem);
```

> **使用场景：** 二进制信号量用于 ISR→任务同步；计数信号量用于资源池管理。

#### 1.3.4 队列 — `osal_queue`

```c
int osal_queue_create(osal_queue_handle_t *queue, uint32_t item_size, uint32_t item_count);
int osal_queue_send(osal_queue_handle_t queue, const void *item, uint32_t timeout_ms);
int osal_queue_receive(osal_queue_handle_t queue, void *item, uint32_t timeout_ms);
```

> 数据按值拷贝传入队列（非指针），确保生产者-消费者解耦。

#### 1.3.5 定时器 — `osal_timer`

```c
int osal_timer_create_periodic(const char *name, osal_timer_cb_t cb, void *arg,
                               uint32_t period_ms, osal_timer_handle_t *timer);
int osal_timer_create_one_shot(const char *name, osal_timer_cb_t cb, void *arg,
                               uint32_t delay_ms, osal_timer_handle_t *timer);
int osal_timer_start(osal_timer_handle_t timer);
int osal_timer_stop(osal_timer_handle_t timer);
```

> 基于 ESP-IDF `esp_timer` 实现，us 级精度。回调在定时器任务上下文执行，不可阻塞。

#### 1.3.6 内存管理 — `osal_memory`

```c
void *osal_malloc_caps(size_t size, uint32_t caps);
void *osal_malloc_internal(size_t size);   // 内部 SRAM
void *osal_malloc_psram(size_t size);       // 外部 PSRAM
void  osal_free(void *ptr);
size_t osal_get_free_heap_size(uint32_t caps);
```

**内存能力标志：**
- `OSAL_MEM_CAP_INTERNAL` — 内部 SRAM（低延迟）
- `OSAL_MEM_CAP_PSRAM` — 外部 PSRAM（大容量）
- `OSAL_MEM_CAP_DMA` — DMA 可访问

> **选型建议：** 帧缓冲、人脸特征等大数据用 PSRAM；ISR 上下文、栈、DMA 描述符等必须用内部 SRAM。

### 1.4 设备驱动层 (DAL)

DAL 在 PAL 之上封装完整的设备驱动，包含外部芯片的初始化序列、模式切换、状态管理。

---

#### 1.4.1 显示 — `dal_display`

**文件：** `components/dal/dal_display/`
**支持芯片：** TC358762 (MIPI DSI→RGB 桥接), ATtiny88 (背光/PWM 管理)

**功能：** 树莓派 7" DSI 显示屏完整驱动链：
- TC358762 初始化 (I2C 寄存器配置 + DSI Generic Write)
- ATtiny88 上电/背光控制 (I2C, 地址 0x45)
- ESP32-P4 DSI 主机启动
- 上电时序管理 (power → backlight 间隔 80ms)

**主要 API：**

```c
int dal_display_init(dal_display_handle_t *handle, const dal_display_config_t *cfg);
int dal_display_deinit(dal_display_handle_t *handle);
int dal_display_on(dal_display_handle_t handle);
int dal_display_off(dal_display_handle_t handle);
int dal_display_draw_bitmap(dal_display_handle_t handle, uint16_t x, uint16_t y,
                            uint16_t w, uint16_t h, const void *data);
int dal_display_fill(dal_display_handle_t handle, uint16_t x, uint16_t y,
                     uint16_t w, uint16_t h, uint32_t color);
int dal_display_get_fb(dal_display_handle_t handle, void **fb0, void **fb1);
int dal_display_set_backlight(dal_display_handle_t handle, uint8_t percent);
```

**配置结构体核心字段：**

| 字段 | 说明 |
|------|------|
| `i2c_bus_handle` | 共用 I2C 总线句柄 |
| `use_attiny88` | 是否使用 ATtiny88 背光管理 |
| `dsi_host / dsi_num_data_lanes / dsi_lane_bit_rate_mbps` | DSI 总线配置 |
| `timing` | DPI 视频时序 (h/v_res, 同步脉宽, 前后肩) |
| `pixel_format / in_color_format` | 像素格式 |
| `num_fbs` | 帧缓冲数 (1=单缓冲, 2=双缓冲) |
| `bl_use_direct_pwm` | 是否使用直接 PWM 背光 (替代 ATtiny88) |

> **上电时序：** ATtiny88 模式→配置→80ms 延迟→DSI 开始扫描。此值是经验值，缩短可能导致显示异常。

#### 1.4.2 触摸 — `dal_touch`

**文件：** `components/dal/dal_touch/`
**支持芯片：** FT5406 (FT5x06 系列电容触摸控制器, I2C 0x38)

**功能：** 树莓派 7" 屏载触摸控制器驱动，支持最多 5 点触控。

**主要 API：**

```c
int dal_touch_init(dal_touch_handle_t *handle, const dal_touch_config_t *cfg);
int dal_touch_deinit(dal_touch_handle_t *handle);
int dal_touch_read(dal_touch_handle_t handle, dal_touch_data_t *data);
```

**数据结构：**

```c
typedef struct {
    uint8_t  id;       /**< 触点 ID (0~4) */
    uint16_t x;        /**< 触点 X 坐标 */
    uint16_t y;        /**< 触点 Y 坐标 */
    uint8_t  event;    /**< 0=按下, 1=抬起, 2=持续 */
} dal_touch_point_t;

typedef struct {
    uint8_t             num_points;     /**< 有效触点数 */
    dal_touch_point_t   points[5];      /**< 触点数据 */
} dal_touch_data_t;
```

> `dal_touch_read()` 为非阻塞调用：返回值 >0 表示触点数，0 表示无触摸，<0 表示错误。

#### 1.4.7 人体红外感应 — `dal_pir`

**文件：** `components/dal/dal_pir/`
**支持传感器：** HC-SR501 人体红外感应模块

**硬件引脚：**

| 引脚 | 连接 | 说明 |
|------|------|------|
| VCC | 3.3V | 传感器供电（若为 5V 版本需电平转换） |
| GND | GND | 共地 |
| OUT | GPIO (可配置) | 数字输出：高电平=有人，低电平=无人 |

> HC-SR501 板上两个调节电位器：**灵敏度**（检测距离 3~7m）、**延时**（移动停止后 OUT 保持高电平时间 2.5s~5min）。触发模式跳线：可重复触发 / 单次触发。

**功能：** 通过 GPIO 双边沿中断检测人体移动，向上层提供运动状态变化回调，用于系统唤醒/休眠调度。

**主要 API：**

```c
int dal_pir_init(dal_pir_handle_t *handle, const dal_pir_config_t *cfg);
int dal_pir_deinit(dal_pir_handle_t *handle);
int dal_pir_set_callback(dal_pir_handle_t handle, dal_pir_state_cb_t cb, void *arg);
int dal_pir_get_state(dal_pir_handle_t handle, dal_pir_state_t *state);
int dal_pir_enable(dal_pir_handle_t handle);
int dal_pir_disable(dal_pir_handle_t handle);
```

**状态与回调：**

```c
typedef enum {
    DAL_PIR_STATE_IDLE   = 0,   /**< 无人（OUT 低电平） */
    DAL_PIR_STATE_MOTION = 1,   /**< 检测到移动（OUT 高电平） */
} dal_pir_state_t;

typedef void (*dal_pir_state_cb_t)(dal_pir_state_t state, void *arg);
```

**配置结构体：**

```c
typedef struct {
    int  gpio_pin;      /**< PIR OUT 引脚 */
    bool pull_down;     /**< 使能内部下拉（推荐 true） */
} dal_pir_config_t;
```

**使用示例（唤醒/休眠调度）：**

```c
/* ---- ISR 回调：通知主任务运动状态变化 ---- */
static void pir_state_changed(dal_pir_state_t state, void *arg)
{
    osal_queue_handle_t *evt_queue = (osal_queue_handle_t *)arg;
    osal_queue_send(evt_queue, &state, 0);
}

/* ---- 主任务：根据 PIR 状态调度唤醒/休眠 ---- */
void app_power_manager(void)
{
    dal_pir_handle_t pir;
    dal_pir_config_t cfg = { .gpio_pin = GPIO_NUM_6, .pull_down = true };
    dal_pir_init(&pir, &cfg);
    dal_pir_set_callback(pir, pir_state_changed, &evt_queue);
    dal_pir_enable(pir);

    dal_pir_state_t state;
    while (1) {
        if (osal_queue_receive(evt_queue, &state, pdMS_TO_TICKS(10000))) {
            if (state == DAL_PIR_STATE_MOTION) {
                system_wake_up();   /* 唤醒屏幕/摄像头 */
            } else {
                system_prepare_sleep();  /* 启动休眠倒计时 */
            }
        }
    }
}
```

**设计要点：**

| 项目 | 说明 |
|------|------|
| 初始化稳定期 | HC-SR501 上电后约 **60 秒**内 OUT 可能不稳定，此期间不应依赖传感器输出 |
| 中断触发 | 双边沿中断 (`ANYEDGE`)：上升沿→有人，下降沿→无人 |
| ISR 安全 | 回调在 ISR 上下文触发（使用 `IRAM_ATTR`），仅做标志位/队列操作，复杂逻辑在任务上下文处理 |
| 内部下拉 | 推荐使能 `pull_down`，防止 OUT 浮空时误触发 |
| 休眠倒计时 | 应用层收到 `IDLE` 后启动定时器（如 30 秒），超时后执行休眠；期间收到 `MOTION` 则取消倒计时 |

**依赖：** `pal_gpio`（GPIO 配置 + 中断注册）

```c
int service_face_feature_get_status(service_face_feature_status_t *status);
int service_face_feature_deinit(service_face_feature_handle_t handle);
```

**设计要点：**

| 项目 | 说明 |
|------|------|
| 人脸对齐 | 优先使用 5 点关键点做仿射对齐，提升跨姿态识别稳定性 |
| 特征格式 | feature 必须带 `model_id` 存入数据库，防止不同模型特征混用 |

#### 1.5.2 服务开发关键决策

本节记录服务层开发时必须统一遵守的设计决策。后续实现服务时，若确实需要偏离以下规则，必须先更新本文档说明原因和影响范围。

##### (1) 服务间通信方式

服务层采用 **事件队列 + 共享数据描述符 + 同步查询 API** 的组合方式，不采用服务之间互相直接调用任务函数的方式。

| 通信类型 | 使用场景 | 推荐机制 | 说明 |
|----------|----------|----------|------|
| 高频数据流 | 摄像头帧、AI 推理输入 | `osal_queue` 传递 frame 描述符 | 队列只传描述符/指针，不复制整帧 |
| 低频事件 | UI 事件、识别结果、告警、状态变化 | 统一 `service_event_t` 事件 | 按值传递，小结构体，便于追踪 |
| 同步查询 | 获取数据库记录、服务状态、统计信息 | `service_xxx_get_status()` / `service_db_xxx()` | 调用方短时间阻塞，接口内部加锁 |
| 控制命令 | start/stop/suspend/resume | 服务公开 API 或控制事件 | APP 层统一编排生命周期 |
| 异常上报 | 外设故障、任务超时、模型不可用 | `service_system_guard_report_fault()` | 所有故障统一进入 guard 和日志服务 |

**核心决策：**

- 摄像头帧不在多个服务间深拷贝，`service_camera` 是 frame buffer 所有权中心。
- AI 推理链路使用队列解耦，摄像头采集不能被检测/识别耗时阻塞。
- UI 更新只能通过 UI 事件队列进入 `service_ui`，其他服务不得直接调用 LVGL 或显示控件。
- 数据库接口可以同步调用，但必须由 `service_db` 内部保证互斥和文件一致性。
- 继电器开门只能由 APP 状态机或权限管理通过明确事件触发，不允许检测服务/识别服务直接控制继电器。

##### (2) 统一事件模型

服务间低频消息统一使用事件结构，事件按值进入队列，避免生命周期不清晰的临时指针。

```c
typedef enum {
    SERVICE_EVT_NONE = 0,
    SERVICE_EVT_CAMERA_FRAME_READY,      /**< 摄像头新帧可用 */
    SERVICE_EVT_FACE_DETECTED,           /**< 检测到人脸 */
    SERVICE_EVT_FACE_IDENTIFIED,         /**< 人脸识别完成 */
    SERVICE_EVT_AUTH_PASS,               /**< 权限校验通过 */
    SERVICE_EVT_AUTH_DENY,               /**< 权限校验拒绝 */
    SERVICE_EVT_TOUCH,                   /**< 触摸输入 */
    SERVICE_EVT_UI_REQUEST,              /**< UI 页面/弹窗请求 */
    SERVICE_EVT_LOG_READY,               /**< 日志待写入/待上传 */
    SERVICE_EVT_MQTT_CONNECTED,          /**< MQTT 已连接 */
    SERVICE_EVT_MQTT_DISCONNECTED,       /**< MQTT 已断开 */
    SERVICE_EVT_FAULT,                   /**< 服务故障 */
} service_event_type_t;

typedef struct {
    service_event_type_t type;       /**< 事件类型 */
    uint32_t             timestamp;  /**< 事件时间戳，Unix time 或系统 tick */
    uint32_t             source;     /**< 事件来源服务 ID */
    uint32_t             arg0;       /**< 小参数 0，如 uid / error_code */
    uint32_t             arg1;       /**< 小参数 1，如 score / state */
    void                *payload;    /**< 可选 payload，必须有明确所有权规则 */
} service_event_t;
```

**payload 使用规则：**

| payload 类型 | 所有权 | 释放方 | 说明 |
|--------------|--------|--------|------|
| `NULL` | 无 | 无 | 首选方式，小数据放 `arg0/arg1` |
| 静态/全局对象 | 生产者 | 不释放 | 只读配置、状态快照可使用 |
| frame 描述符 | `service_camera` | `service_camera` | 消费者不得释放底层 buffer |
| 动态申请对象 | 事件消费者 | 消费者处理后释放 | 必须在事件类型文档中明确 |

> 原则：除摄像头帧外，服务事件尽量不携带指针。需要大数据传输时应建立专用队列或资源池。

##### (3) 任务与队列划分

服务层按“实时性”和“阻塞风险”划分任务，避免一个任务承担多个不同实时等级的工作。

| 任务 | 归属服务 | 优先级建议 | 队列深度 | 说明 |
|------|----------|------------|----------|------|
| `camera_task` | `service_camera` | 高 | AI=2, UI=1 | 只负责取帧和分发，不做推理 |
| `face_detect_task` | `service_face_detect` | 中高 | 2 | 取最新帧做人脸检测，慢时丢旧帧 |
| `face_identify_task` | `service_face_identify` | 中 | 2 | 特征提取和 1:N 比对 |
| `ui_task` | `service_ui` | 中 | 8~16 | 唯一允许调用 UI 框架的任务 |
| `touch_task` | `service_touch` | 中 | 8 | 周期读取触摸并转成 UI 事件 |
| `log_task` | `service_log` | 低 | 16~32 | 批量落盘和导出，关键日志可同步刷盘 |
| `mqtt_task` | `service_mqtt` | 低 | 16 | 网络发布/订阅，断线重连 |
| `guard_timer/task` | `service_system_guard` | 高 | 无或 8 | 心跳检查和故障升级 |

**队列策略：**

- 摄像头 → AI：队列满时丢弃旧帧，保留最新帧，保证识别结果尽量实时。
- 摄像头 → UI：预览允许降帧，UI 不得反压摄像头采集。
- 触摸 → UI：不丢弃按下/抬起事件；滑动 move 事件允许合并。
- 识别结果 → APP/UI/日志：不允许静默丢弃，队列满时记录 guard warning。
- MQTT 上报：允许离线缓存，不能阻塞本地识别和开门流程。

##### (4) 摄像头帧所有权决策

`service_camera` 统一管理 `dal_camera_frame_t` 的生命周期。

```text
dal_camera_capture_frame()
    → service_camera 包装为 service_camera_frame_msg_t
    → 投递到 UI / AI 队列
    → 等待消费者处理完成或超时
    → dal_camera_release_frame()
```

建议帧消息结构：

```c
typedef struct {
    uint32_t frame_id;                 /**< service_camera 分配的帧 ID */
    dal_camera_frame_t frame;          /**< DAL 帧描述符，底层 buffer 不复制 */
    uint32_t ref_count;                /**< 消费者引用计数 */
    uint32_t flags;                    /**< 预览/AI/调试标志 */
} service_camera_frame_msg_t;
```

**决策说明：**

- `dal_camera_release_frame()` 只能由 `service_camera` 调用。
- 消费者只读取 `frame.data`，不得保存超过本次处理周期。
- AI 任务处理超时时，`service_camera` 可以回收旧帧并记录一次丢帧计数。
- 若后续需要长期保存图片，必须复制到独立 buffer 或写入 SD 卡文件，不能持有 V4L2 mmap buffer。

##### (5) 服务生命周期决策

服务初始化由 APP 层统一编排，顺序遵循“底层资源先就绪，业务服务后启动”。

```text
PAL/OSAL init
    → DAL init: storage / network / display / touch / camera / relay / pir
    → service_db_init()
    → service_log_init()
    → service_face_detect_init()
    → service_face_identify_init()
    → service_user_manage_init()
    → service_ui_init()
    → service_touch_init()
    → service_camera_init()
    → service_mqtt_init()
    → service_system_guard_init()
    → start 各服务任务
```

**启动/停止规则：**

| 场景 | 处理 |
|------|------|
| 进入 WORKING | 启动 display/camera/touch/AI，恢复 UI 页面 |
| 进入 DEEP_SLEEP | 停止 camera/AI/touch/UI tick，关闭背光，保留 PIR 唤醒 |
| 识别模型缺失 | 服务可启动但 `model_ready=false`，禁止真实开门 |
| SD 卡异常 | 禁止用户录入和日志导出；已有 RAM 特征库可短时只读识别 |
| 网络异常 | MQTT 停止或重连，本地识别继续运行 |
| 发生 FATAL | 停止业务任务，记录故障，安全关闭继电器，准备复位 |

##### (6) 线程安全与互斥决策

| 资源 | 保护方式 | 说明 |
|------|----------|------|
| `service_db` 文件和索引 | 服务内部 mutex | 所有 DB API 内部加锁 |
| 人脸特征库 RAM 缓存 | RW 语义或普通 mutex | 识别读多写少，录入/删除时暂停匹配或加写锁 |
| UI 对象 | UI 任务单线程 | 非 UI 任务只发事件 |
| 摄像头设备 | `service_camera` 单所有者 | 其他服务不得直接调用 `dal_camera_capture_frame()` |
| MQTT client | `mqtt_task` 单所有者 | 发布请求通过队列进入 MQTT 任务 |
| 继电器 | APP 状态机单点控制 | 权限服务只给出 allowed，不直接吸合 |

**锁使用规则：**

- 禁止在持有 DB mutex 时执行 AI 推理、网络发送、UI 刷新等长耗时操作。
- 禁止跨服务嵌套持锁；如必须访问多个服务，先复制所需数据再释放锁。
- ISR 上下文不允许直接调用服务 API，必须通过任务通知或队列转发。

##### (7) 错误处理与降级决策

| 故障 | 降级策略 | 是否允许开门 |
|------|----------|--------------|
| 检测模型缺失 | placeholder 只用于流程调试 | 否 |
| 特征模型缺失 | 返回 `MODEL_NOT_READY` | 否 |
| 特征库为空 | UI 提示未录入用户 | 否 |
| 摄像头连续超时 | 重启 stream，失败进入 ERROR | 否 |
| SD 卡不可写 | 禁止录入，日志进入 RAM 缓存 | 可按已加载特征短时运行 |
| MQTT 断线 | 本地缓存事件，后台重连 | 是 |
| UI 异常 | 可进入无屏识别模式，日志记录 | 是，需本地策略允许 |
| guard FATAL | 关闭继电器并准备复位 | 否 |

**安全底线：**

- 任何 placeholder / mock / debug 结果不得触发真实继电器动作。
- 任何模型状态为 `model_ready=false` 的识别结果不得设置 `matched=true`。
- 允许开门必须经过 `service_user_manage_check_permission()` 或等价权限校验路径。
- 所有拒绝开门原因都应写入日志，便于现场排查。

##### (8) 配置与持久化决策

| 配置类型 | 存储位置 | 说明 |
|----------|----------|------|
| 编译期硬件配置 | `dal_config.h` | 引脚、默认模型路径、默认阈值 |
| 运行期系统配置 | `/sdcard/config/system.json` | MQTT、显示亮度、识别阈值、设备 ID |
| 用户/权限数据 | `service_db` users.jsonl | 用户名、PIN、权限、有效期 |
| 人脸特征 | `service_db` face_index/data/journal | 带 `model_id`，支持掉电恢复 |
| 通行日志 | `service_db` 月度 bin 文件 | 带 CRC32，支持查询/清理 |
| 导出文件 | `/sdcard/export/` | CSV 日志、调试抓拍、诊断包 |

### 1.7 应用层 (APP)

**文件：** `main/main.c`

**功能：** 系统初始化与业务流程编排。当前实现了基于 PIR 事件驱动的状态机框架：

```
app_main()
  │
  ├─ [1] dal_pir_init()              → PIR 传感器初始化 + 回调注册
  ├─ [2] dal_pir_enable()            → 使能 PIR 中断检测
  ├─ [3] osal_task_create()          → 创建 System_Manager_Task
  └─ [4] System_Manager_Task()       → 事件驱动状态机主循环
```

**系统状态机：**

```c
typedef enum {
    SYS_STATE_DEEP_SLEEP,    // 深度休眠：仅 PIR 中断工作，极低功耗
    SYS_STATE_IDLE,          // 待机缓冲：PIR 触发后等待确认 / 人刚离开倒计时
    SYS_STATE_WORKING,       // 核心工作：摄像头采集、AI 鉴权、UI 交互全开
    SYS_STATE_RELAY_OPEN,    // 开门执行：继电器吸合，倒计时准备关门
    SYS_STATE_ERROR,         // 故障态：外设异常或看门狗复位前的安全状态
} system_state_t;
```

**核心任务 — System_Manager_Task：**

```c
void System_Manager_Task(void *pvParameters)
{
    system_state_t current_state = SYS_STATE_DEEP_SLEEP;

    for (;;) {
        // 阻塞等待事件通知（最多 1 秒，用于超时检查）
        if (xTaskNotifyWait(0, 0xFFFFFFFF, &ulNotifyValue, pdMS_TO_TICKS(1000)) == pdTRUE) {
            switch (ulNotifyValue) {
                case EVT_PIR_RISING:    // 人靠近 → 唤醒系统
                case EVT_AUTH_SUCCESS:  // 鉴权成功 → 开门
                case EVT_RELAY_CLOSED:  // 门已关 → 进入待机
            }
        }

        // IDLE 态超时 15 秒无事件 → 进入深度休眠
        if (current_state == SYS_STATE_IDLE && timeout_expired) {
            Suspend_All_Business_Tasks();
            current_state = SYS_STATE_DEEP_SLEEP;
        }
    }
}
```

**PIR 回调 → 事件通知桥接：**

```
HC-SR501 OUT 电平变化
    → dal_pir ISR (IRAM_ATTR)
    → dal_pir_state_cb_t 回调
    → xTaskNotifyGive() / xTaskNotify()
    → System_Manager_Task 唤醒处理
```

**状态说明：**

| 状态 | 显示 | 摄像头 | PIR | 说明 |
|------|------|--------|-----|------|
| DEEP_SLEEP | 关 | 关 | 中断待命 | 极低功耗，仅 PIR GPIO 中断可唤醒 |
| IDLE | 开 | 开 | 监测中 | PIR 触发后的待机缓冲态，或人刚离开的倒计时缓冲（15s） |
| WORKING | 开 | 开 | 监测中 | 核心识别状态：摄像头采集 + AI 鉴权 + UI 交互 |
| RELAY_OPEN | 开 | 关 | 监测中 | 识别通过，继电器吸合开门 |

#### 1.7.2 主要事件流

| 事件 | 触发源 | 处理 |
|------|--------|------|
| EVT_PIR_RISING | HC-SR501 PIR 上升沿中断 | 唤醒系统：`Wake_Up_System()` → 开屏 + 使能摄像头 |
| EVT_AUTH_SUCCESS | AI 鉴权通过 | `Open_Relay()` → 继电器吸合开门 |
| EVT_RELAY_CLOSED | 继电器超时/门关闭反馈 | 回到 IDLE 态 |
| PIR 无人超时 | IDLE 态 15 秒无新事件 | `Suspend_All_Business_Tasks()` → 进入 DEEP_SLEEP |
| 人脸检测到 | ISP 输出帧 + AI 推理 | 特征提取 → 1:N 比对 |
| 识别成功 | 比对分数 > 阈值 | 发送 EVT_AUTH_SUCCESS + 记录日志 |
| 识别失败 | 比对分数 < 阈值 | 记录拒绝日志 |
| 触摸输入 | FT5406 中断/轮询 | UI 事件分发 |
| 网络断开 | ETH PHY 状态变化 | 本地缓存 + 重连 |
| 胁迫事件 | 特定 PIN/手势 | 静默报警 + 日志标记 |
| 外设异常 | 硬件故障/超时 | 进入 ERROR 态 |

## 3 开发规范

### 3.1 代码风格

详见项目根目录 `CLAUDE.md`（全局规则）。关键要点：

- **注释语言：** 中文 (Doxygen 格式)，代码符号保持英文
- **函数命名：** `module_verbnoun()` 如 `dal_display_init()`
- **返回约定：** 0 = 成功，<0 = 错误码
- **ISR 约束：** 仅设标志位/发队列，不做耗时操作
- **共享变量：** `volatile` 修饰，多字节访问需临界区保护

### 3.2 新增模块指引

**新增 PAL 模块：**

1. 在 `components/pal/` 下创建 `pal_<外设>/` 目录
2. 头文件放 `include/`，源文件放模块根目录
3. 遵循统一接口约定：`init/deinit` 配对，所有函数返回 `int` 错误码
4. 在 `components/pal/CMakeLists.txt` 中添加编译目标
5. 更新本文档 PAL 章节

**新增 DAL 模块：**

1. 在 `components/dal/` 下创建 `dal_<设备>/` 目录
2. 使用 `pal_*_handle_t` 接收底层资源
3. 封装芯片特定初始化序列
4. 对外暴露 `dal_<device>_init/deinit` 及功能 API

**新增服务模块：**

1. 在 `components/service/` 下创建对应目录
2. 依赖 DAL 层接口，禁止直接调用 PAL/ESP-IDF
3. 所有持久化数据通过 `dal_storage` 提供的挂载点
4. 外部 API 在独立 `include/` 头文件中声明

### 3.3 错误处理

```c
// PAL/OSAL/DAL 统一使用返回码模式
int ret = dal_display_init(&display, &cfg);
if (ret != 0) {
    PAL_LOGE(TAG, "显示初始化失败: %d", ret);
    // 逐级回滚已分配资源
    goto cleanup;
}
```

> **禁止在库代码中使用 `assert()` 处理运行时可恢复错误。** `assert` 仅用于"不可能发生"的编程逻辑错误。

### 3.4 内存使用原则

| 数据类型 | 推荐位置 | 原因 |
|----------|----------|------|
| 任务栈 | 内部 SRAM | 低延迟访问 |
| DMA 描述符 | 内部 SRAM | 硬件要求 |
| ISR 上下文 | 内部 SRAM | 禁止 PSRAM 访问 |
| 帧缓冲 | PSRAM | 大容量 (~2MB) |
| 人脸特征库 | PSRAM | 可动态加载/卸载 |
| 数据库索引 | 内部 SRAM | 常驻内存 |

### 4.3 调试技巧

- **PIR 状态监控：** `dal_pir_get_state()` 可在任务上下文随时读取当前运动状态，用于排查中断是否正常触发
- **GPIO 电平验证：** 用万用表/逻辑分析仪测量 HC-SR501 OUT 引脚电平，确认硬件输出与软件状态一致
- **中断调试：** 在 `dal_pir_isr_handler` 中添加 GPIO 翻转（而非 printf），用逻辑分析仪测量中断响应时间
- **Cache 一致性：** 花屏/撕裂通常由 cache 未同步引起，检查 `esp_cache_msync()` 调用方向：
  - ISP DMA → CPU 读：`ESP_CACHE_MSYNC_FLAG_DIR_M2C`
  - CPU 写 → DSI DMA：`ESP_CACHE_MSYNC_FLAG_DIR_C2M`
- **I2C 冲突：** 共用 I2C 总线的设备（OV5647 + ATtiny88）操作前确保互斥
- **栈溢出：** 使用 `osal_task_get_stack_high_water_mark()` 监控最小栈余量
- **状态机调试：** 在 `System_Manager_Task` 的事件处理分支添加 `PAL_LOGI` 输出当前状态和事件值，跟踪状态迁移是否正确

#### 4.4.5 安全约束

- 模型文件不存在时，系统允许启动，但识别服务必须报告 `model_ready=false`。

- 真实开门必须满足：检测模型 ready、识别模型 ready、特征库 ready、匹配分数超过现场标定阈值。

### 4.5 常见问题

| 现象 | 可能原因 | 排查方向 |
|------|----------|----------|
| 摄像头无图 | I2C 通信失败 / SCCB 未初始化 | 检查 GPIO7/8 引脚、I2C probe 0x36 |
| 花屏/撕裂 | Cache 未同步 / DSI 时序错误 | 确认 `esp_cache_msync` 方向、DPI 时钟值 |
| 背光不亮 | ATtiny88 无响应 | I2C probe 0x45、上电时序间隔 |
| 帧率低 | ISP→memcpy 开销 | 考虑 DMA2D 代替 CPU memcpy |
| SD 卡读写失败 | 引脚复用冲突 / 供电不足 | 检查 SDMMC 引脚、供电电压 |
| PIR 频繁误触发 | OUT 浮空 / 电源噪声 | 使能 GPIO 内部下拉，VCC 加 100nF 去耦电容 |
| PIR 上电后一直高电平 | 未过初始化稳定期 | 等待 **60 秒**稳定期后读取状态 |
| PIR 无法唤醒系统 | GPIO 中断未使能 | 确认 `dal_pir_enable()` 已调用，引脚号正确 |
| 休眠后无法唤醒 | 电平转换电路漏电 / GPIO 配置丢失 | 检查休眠时 GPIO 保持输入+中断使能状态 |
